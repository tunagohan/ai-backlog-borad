# Experiments (Data / Validation)

## Hypotheses
- H1: 複数物件オーナーにとって最大の痛みは「点検実施そのもの」より「報告確認と履歴追跡の負荷」である
- H2: 紙/Excel運用からでも、現場担当はモバイル入力に移行できる（運用負荷が下がる）
- H3: 「物件階層での履歴可視化」が導入判断の主要要因になり、価格受容性を押し上げる
- H4: 初期獲得は広告より紹介・直接提案の方が反応率/CACが良い

## Experiments
- (例) Zenn記事 + waitlist → 登録率/反応
- (例) 5名インタビュー → pain score
- (例) 有料β → 決済率
- E1 (課題検証): 8〜12名インタビュー（オーナー/運営責任者/点検担当）
  - 対応仮説: H1, H2
  - 取得方法: 半構造化インタビュー + 現行運用フローの時系列ヒアリング
- E2 (ソリューション検証): クリック可能プロトタイプを使ったタスク観察（5社）
  - 対応仮説: H2, H3
  - 取得方法: 「点検入力→不具合報告→管理者確認」の完了率と所要時間を計測
- E3 (価格検証): 料金感度ヒアリング + コンジョイント簡易比較（10社）
  - 対応仮説: H3
  - 取得方法: 3価格帯（低/中/高）提示時の導入意向と阻害要因を記録
- E4 (獲得チャネル検証): 4週間で2チャネルABテスト（紹介/コンテンツ）
  - 対応仮説: H4
  - 取得方法: LP流入、問い合わせ、商談化のファネルをチャネル別に計測

## Metrics
- North Star（探索期）:
  - 検証対象ICPにおける「週次/月次報告確認時間の削減見込み（自己申告）」 
- 補助指標:
  - インタビュー pain score（0〜10）
  - プロトタイプタスク完了率（%）/完了時間（分）
  - 有償導入意向率（%）/支払い意思額レンジ
  - チャネル別 CVR（LP→問い合わせ→商談）

## Success / Failure Criteria
- H1 判定:
  - Success: インタビュー対象の70%以上が「確認/追跡負荷」を最重要課題Top2に挙げる
  - Failure: 50%未満、または主課題が別領域（契約/請求/労務）に集中
- H2 判定:
  - Success: プロトタイプで現場タスク完了率80%以上、紙運用比で時間短縮実感が多数
  - Failure: 完了率60%未満、または入力負荷の増加指摘が多数
- H3 判定:
  - Success: 料金提示後も50%以上が「試験導入したい」と回答
  - Failure: 25%未満、または価格より機能不足が主要理由
- H4 判定:
  - Success: 紹介チャネルの商談化率が他チャネル比1.5倍以上
  - Failure: 有意差なし、または紹介依存で再現性が低い
